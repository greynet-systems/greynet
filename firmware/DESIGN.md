# Firmware Design: Умная коробка Greynet

## User Stories (в порядке разработки)

### US-1: Коробка получает интернет
**Как** пользователь,
**я хочу** воткнуть коробку в любой LAN-порт роутера,
**чтобы** она автоматически получила IP и доступ в интернет.

**Acceptance criteria:**
- eth0 настроен как WAN (DHCP client)
- После загрузки коробка имеет IP и default route
- Проверка: `curl http://cp.cloudflare.com/generate_204` → 204

**Компоненты:** OpenWrt network config, health-check скрипт

---

### US-2: Коробка определяет блокировки
**Как** коробка,
**я хочу** понять какие типы DPI-блокировок применяет провайдер,
**чтобы** подобрать правильную стратегию обхода.

**Acceptance criteria:**
- Запускается dpi-detector, определяет типы блокировок (TLS SNI, TCP throttling, UDP/QUIC)
- Результат сохраняется в `/etc/greynet/dpi-report.json`
- При необходимости запускается zapret2 blockcheck для точечного подбора
- Найденная стратегия сохраняется в `/etc/greynet/strategy.conf`

**Компоненты:** dpi-detector (Docker/binary), zapret2, strategies DB

---

### US-3: Коробка обходит блокировки
**Как** коробка,
**я хочу** применить найденную стратегию обхода DPI,
**чтобы** исходящий трафик не блокировался.

**Acceptance criteria:**
- zapret2 запущен с подобранной стратегией (NFQWS2_OPT)
- Проверка: заблокированные домены доступны через curl с коробки
- Сервис автостартует при загрузке

**Компоненты:** zapret2 service, iptables/nftables rules

---

### US-4: Коробка раздаёт прокси
**Как** коробка,
**я хочу** поднять HTTP/SOCKS5 прокси,
**чтобы** клиентские устройства могли направлять трафик через меня.

**Acceptance criteria:**
- Прокси слушает на Tailscale интерфейсе (не на WAN!)
- Исходящий трафик прокси проходит через zapret2 (NFQUEUE)
- Проверка: curl через прокси к заблокированному домену → 200

**Компоненты:** tinyproxy / microsocks / 3proxy, iptables маркировка

---

### US-5: Коробка подключается к mesh-сети
**Как** коробка,
**я хочу** зарегистрироваться в Headscale mesh-сети,
**чтобы** клиенты могли подключаться ко мне из любой точки.

**Acceptance criteria:**
- Tailscale client установлен и подключён к Headscale серверу
- Коробка получает стабильный IP в mesh-сети (100.x.x.x)
- Коробка доступна по Tailscale IP с других устройств в сети

**Компоненты:** tailscale client (OpenWrt), headscale server (отдельный хост/VPS)

---

### US-6: Пользователь подключается по QR-коду
**Как** пользователь,
**я хочу** отсканировать QR-код,
**чтобы** моё устройство автоматически настроилось на работу через коробку.

**Acceptance criteria:**
- Коробка генерирует QR-код с данными подключения
- QR содержит: Headscale URL + auth key + proxy endpoint
- После сканирования устройство подключается к mesh и настраивает прокси
- QR доступен через web-интерфейс на коробке (eth1 / LAN)

**Компоненты:** QR генератор, мини web-UI, headscale preauthkey API

---

### US-7: Коробка мониторит и адаптируется
**Как** коробка,
**я хочу** периодически проверять работоспособность обхода,
**чтобы** при смене стратегии ISP автоматически подобрать новую.

**Acceptance criteria:**
- Крон-задача проверяет доступ к контрольным доменам (раз в час)
- При обнаружении блокировки: dpi-detector → blockcheck → apply → verify
- Логи диагностики доступны через web-UI / API

**Компоненты:** cron/procd, watchdog скрипт, health API

---

## Порядок разработки

```
US-1 (интернет) → US-2 (детект) → US-3 (обход) → US-4 (прокси)
                                                       ↓
                              US-7 (мониторинг) ← US-6 (QR) ← US-5 (mesh)
```

US-1..3 — ядро, можно разрабатывать и тестировать на коробке прямо сейчас.
US-4..5 — сетевой слой, требует поднятый Headscale сервер.
US-6..7 — UX и надёжность, финальная полировка.
